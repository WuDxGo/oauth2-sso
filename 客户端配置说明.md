# OAuth2 客户端配置字段说明

## 基本字段

### 1. 客户端 ID (client_id)
- **作用**: 客户端的唯一标识符
- **示例**: `gateway-client`, `order-service`
- **说明**: 
  - 类似用户名，用于标识是哪个应用
  - 创建后不可修改
  - 只能包含字母、数字、下划线和短横线
  - 在 OAuth2 请求中会携带此参数

### 2. 客户端名称 (client_name)
- **作用**: 客户端的显示名称
- **示例**: `网关客户端`, `订单服务`
- **说明**: 
  - 用于管理后台显示
  - 可以随时修改
  - 帮助用户识别是哪个应用

### 3. 客户端密钥 (client_secret)
- **作用**: 客户端的认证密码
- **示例**: `gateway-secret`, `order-secret`
- **说明**: 
  - 与 client_id 配合使用进行认证
  - 类似登录密码，需要妥善保管
  - 泄露后应立即修改
  - 编辑时留空表示不修改

## 认证方式

### client_secret_basic
- **说明**: 通过 HTTP Basic Auth 传递密钥
- **请求示例**:
  ```http
  Authorization: Basic base64(client_id:client_secret)
  ```
- **适用场景**: 大多数场景（推荐）
- **安全性**: ⭐⭐⭐⭐⭐

### client_secret_post
- **说明**: 通过 POST 表单参数传递密钥
- **请求示例**:
  ```
  client_id=xxx&client_secret=yyy
  ```
- **适用场景**: 不支持 Basic Auth 的客户端
- **安全性**: ⭐⭐⭐⭐

## 授权模式 (Grant Types)

### authorization_code (授权码模式)
- **作用**: 最安全的授权模式，用于 Web 应用
- **流程**: 
  1. 用户跳转到认证页面登录
  2. 认证成功后返回授权码
  3. 用授权码换取 Token
- **适用场景**: 有后端的 Web 应用
- **安全性**: ⭐⭐⭐⭐⭐
- **是否需要重定向 URI**: ✅ 是

### password (密码模式)
- **作用**: 直接用用户名密码换取 Token
- **流程**: 
  1. 用户输入用户名密码
  2. 客户端直接发送给认证服务器
  3. 返回 Token
- **适用场景**: 可信应用（如网关、官方 App）
- **安全性**: ⭐⭐⭐
- **注意**: 不建议用于第三方应用

### client_credentials (客户端凭证模式)
- **作用**: 客户端用自己的身份认证，不涉及用户
- **流程**: 
  1. 客户端发送 client_id 和 client_secret
  2. 返回客户端自己的 Token
- **适用场景**: 服务间调用（微服务）
- **安全性**: ⭐⭐⭐⭐
- **典型用例**: 
  - 订单服务调用用户服务
  - 后台任务访问 API

### refresh_token (刷新 Token)
- **作用**: 用于获取新的访问 Token
- **说明**: 
  - 通常不单独使用，与其他模式配合
  - Access Token 过期后，用 Refresh Token 换取新的 Access Token
  - 避免用户频繁登录
- **有效期**: 通常比 Access Token 长（如 7 天）

## 授权范围 (Scopes)

### openid
- **作用**: 启用 OpenID Connect (OIDC)
- **说明**: 获取用户的身份信息（ID Token）
- **适用场景**: 单点登录 (SSO)

### profile
- **作用**: 获取用户基本信息
- **包含字段**: 用户名、昵称、头像等
- **适用场景**: 需要显示用户信息的场景

### read
- **作用**: 读取权限
- **说明**: 只能读取数据，不能修改
- **适用场景**: 查看数据的应用

### write
- **作用**: 写入权限
- **说明**: 可以创建、修改、删除数据
- **适用场景**: 需要操作数据的应用

---

## 其他字段

### 重定向 URI (redirect_uris)
- **作用**: 授权成功后跳转的地址
- **示例**: `http://localhost:8081/login/oauth2/code/gateway-client`
- **说明**: 
  - 用于授权码模式和隐式模式
  - 必须与配置完全匹配（包括协议、端口）
  - 可以配置多个，用逗号分隔
  - **重要**: 生产环境必须使用 HTTPS

### 需要授权同意 (require_consent)
- **作用**: 控制用户首次授权是否需要手动同意
- **选项**:
  - ✅ 是：用户需要点击"同意"按钮
  - ❌ 否：自动授权，无感知
- **适用场景**:
  - 是：第三方应用、敏感权限
  - 否：内部应用、可信应用

## 配置示例

### 示例 1：网关客户端（前端用户登录）
```yaml
客户端 ID: gateway-client
客户端名称：网关客户端
客户端密钥：gateway-secret
认证方式：client_secret_basic
授权模式：authorization_code, password, refresh_token
重定向 URI: http://localhost:8081/login/oauth2/code/gateway-client
授权范围：openid, profile, read, write
需要授权同意：否
```

### 示例 2：订单服务（服务间调用）
```yaml
客户端 ID: order-service
客户端名称：订单服务
客户端密钥：order-secret
认证方式：client_secret_basic
授权模式：client_credentials
授权范围：read, write
需要授权同意：是
```

### 示例 3：第三方应用
```yaml
客户端 ID: third-party-app
客户端名称：第三方合作伙伴应用
客户端密钥：third-party-secret
认证方式：client_secret_basic
授权模式：authorization_code, refresh_token
重定向 URI: https://partner.com/oauth2/callback
授权范围：openid, profile, read
需要授权同意：是
```

## 安全建议

1. **密钥管理**
   - 生产环境使用强密码（至少 16 位）
   - 定期更换密钥
   - 不要将密钥提交到代码仓库

2. **最小权限原则**
   - 只授予必要的 Scope
   - 服务间调用不需要 openid/profile

3. **重定向 URI**
   - 生产环境必须使用 HTTPS
   - 精确匹配，不要使用通配符

4. **授权模式选择**
   - 优先使用 authorization_code
   - 避免使用 password 模式（除非完全可信）
   - 服务间调用使用 client_credentials

5. **监控审计**
   - 记录客户端的创建、修改、删除操作
   - 定期检查长期未使用的客户端
